# 校园活动策划与报名管理平台 Prototype Notes

## 1. 文档定位

本文件是 `Requirements Agent` 的正式输入之一，用于描述最终系统的页面结构、核心交互、状态流转和原型意图。

- 本文件描述的是目标系统原型，不等同于当前 `index.html` 的现状
- `index.html` 仅作为低保真参考，说明最小闭环已经验证
- 当 `index.html` 与本文件存在差异时，以本文件为后续需求提取依据

## 2. 原型目标

原型需要支撑三个主要角色的完整操作闭环：

1. 学生：发现活动、查看详情、报名、取消报名、查看我的报名
2. 活动负责人：创建活动、编辑活动、提交审核、查看审核结果、管理报名名单
3. 管理员：审核活动、发布活动、驳回活动、查看平台统计

原型强调的是信息架构和业务流程，不要求在此阶段给出高保真视觉设计。

## 3. 信息架构

系统应至少包含以下页面或功能视图：

1. 首页 / 活动广场
2. 登录页
3. 注册页
4. 活动详情页
5. 学生个人中心
6. 活动负责人工作台
7. 创建 / 编辑活动页
8. 管理员审核中心
9. 平台统计页或后台概览页

## 4. 页面说明

### 4.1 首页 / 活动广场

#### 页面目标

为游客和学生提供统一的活动浏览入口。

#### 主要模块

1. 顶部导航
2. 搜索框
3. 活动类型筛选
4. 活动状态筛选
5. 时间区间筛选
6. 活动卡片列表
7. 登录 / 注册入口
8. 不同角色的工作台入口

#### 活动卡片信息

每张活动卡片至少展示：

1. 活动名称
2. 活动类型
3. 活动时间
4. 活动地点
5. 主办方
6. 当前状态
7. 已报名人数 / 人数上限
8. 报名截止时间
9. 详情入口

#### 页面交互

1. 输入关键词可按活动名称、地点、主办方进行搜索。
2. 可按类型、状态、时间筛选活动。
3. 点击活动卡片或“查看详情”进入活动详情页。
4. 游客可以浏览公开活动，但报名按钮应引导其先登录。

### 4.2 登录页

#### 页面目标

提供统一身份入口，支持不同角色登录系统。

#### 主要字段

1. 账号标识：学号、工号或用户名
2. 密码
3. 登录按钮
4. 跳转注册入口

#### 关键交互

1. 登录成功后，根据角色跳转到相应首页或工作台。
2. 登录失败时给出明确错误提示。

### 4.3 注册页

#### 页面目标

提供基础注册能力，使学生和活动负责人能够进入系统。

#### 主要字段

1. 用户名
2. 学号 / 工号
3. 密码
4. 确认密码
5. 角色选择
6. 联系方式

#### 关键交互

1. 注册成功后允许用户直接登录。
2. 不允许重复注册相同账号标识。

### 4.4 活动详情页

#### 页面目标

承载活动完整信息展示和报名入口。

#### 页面内容

1. 活动名称
2. 活动类型
3. 活动简介
4. 主办方
5. 活动开始 / 结束时间
6. 报名截止时间
7. 活动地点
8. 参与对象
9. 人数上限
10. 已报名人数
11. 活动状态
12. 报名按钮 / 取消报名按钮

#### 页面交互

1. 已登录学生可直接报名。
2. 已报名学生可在允许条件下取消报名。
3. 活动已满员、已截止、未发布、已取消时，不应显示可报名状态。

### 4.5 学生个人中心

#### 页面目标

让学生查看和管理自己的活动参与情况。

#### 主要模块

1. 我的资料
2. 我的报名活动列表
3. 报名状态
4. 可取消报名入口

#### 列表字段

1. 活动名称
2. 活动时间
3. 活动地点
4. 当前状态
5. 报名时间
6. 操作按钮

### 4.6 活动负责人工作台

#### 页面目标

为活动负责人提供活动管理入口。

#### 主要模块

1. 我创建的活动列表
2. 活动状态统计
3. 创建新活动入口
4. 草稿活动管理
5. 已提交审核活动管理
6. 已发布活动报名名单入口

#### 列表字段

1. 活动名称
2. 活动状态
3. 活动时间
4. 报名人数
5. 最近更新时间
6. 编辑 / 提交审核 / 查看报名名单操作

### 4.7 创建 / 编辑活动页

#### 页面目标

用于活动负责人录入和维护活动信息。

#### 表单字段

1. 活动名称
2. 活动类型
3. 活动简介
4. 主办方
5. 活动地点
6. 活动开始时间
7. 活动结束时间
8. 报名截止时间
9. 人数上限
10. 面向对象
11. 联系方式
12. 注意事项

#### 页面交互

1. 可以保存为草稿。
2. 可以从草稿提交审核。
3. 已驳回活动可修改后再次提交。
4. 已发布活动若修改关键字段，应触发重新审核。

### 4.8 管理员审核中心

#### 页面目标

为管理员提供标准化审核入口。

#### 主要模块

1. 待审核活动列表
2. 审核详情面板
3. 审核通过按钮
4. 审核驳回按钮
5. 驳回原因输入框
6. 历史审核记录查看

#### 列表字段

1. 活动名称
2. 发起人
3. 提交时间
4. 活动时间
5. 活动地点
6. 当前状态

#### 页面交互

1. 管理员打开活动详情查看完整信息。
2. 对信息完整、内容合理的活动执行通过。
3. 对存在问题的活动执行驳回，并填写原因。
4. 审核完成后，状态和记录应立即更新。

### 4.9 平台统计页 / 后台概览页

#### 页面目标

为管理员提供平台运营视图。

#### 主要信息

1. 活动总数
2. 待审核活动数
3. 已发布活动数
4. 已完成活动数
5. 总报名人次
6. 最近报名记录
7. 最近审核记录

## 5. 核心流程说明

### 5.1 游客浏览活动流程

1. 游客进入首页。
2. 浏览已发布活动列表。
3. 通过搜索和筛选缩小范围。
4. 进入活动详情页查看完整信息。
5. 若尝试报名，则引导登录。

### 5.2 学生注册与登录流程

1. 学生进入注册页填写必要信息。
2. 系统校验信息合法性并完成注册。
3. 学生登录成功后进入学生视图。

### 5.3 学生报名流程

1. 学生在活动详情页点击报名。
2. 系统校验活动状态、剩余名额、是否重复报名、是否已截止。
3. 校验通过则创建报名记录。
4. 系统更新已报名人数。
5. 页面反馈报名成功。

### 5.4 学生取消报名流程

1. 学生在个人中心或活动详情页点击取消报名。
2. 系统校验当前是否允许取消。
3. 若允许，则删除或失效该报名记录。
4. 系统回收名额并更新人数。

### 5.5 活动负责人创建并提交审核流程

1. 活动负责人进入工作台。
2. 点击创建活动。
3. 填写表单并保存草稿或直接提交审核。
4. 提交成功后活动状态变为 `pending_review`。

### 5.6 管理员审核通过流程

1. 管理员打开待审核活动。
2. 核查关键信息。
3. 点击审核通过。
4. 活动状态更新为 `published`。
5. 活动在活动广场展示，并开放报名。

### 5.7 管理员驳回流程

1. 管理员打开待审核活动。
2. 发现信息缺失或内容不合理。
3. 填写驳回原因并执行驳回。
4. 活动状态更新为 `rejected`。
5. 活动负责人在工作台查看驳回原因并修改后重新提交。

## 6. 状态设计

### 6.1 活动状态

活动至少有如下状态：

1. `draft`
2. `pending_review`
3. `rejected`
4. `published`
5. `registration_closed`
6. `completed`
7. `cancelled`

### 6.2 状态流转

```text
draft -> pending_review
pending_review -> published
pending_review -> rejected
rejected -> pending_review
published -> registration_closed
published -> cancelled
registration_closed -> completed
```

### 6.3 报名状态

报名记录至少应支持：

1. `registered`
2. `cancelled`

## 8. 异常与边界场景

原型需覆盖或至少在需求中显式体现以下场景：

1. 游客尝试报名
2. 用户重复报名同一活动
3. 活动满员后继续报名
4. 报名截止后继续报名
5. 非管理员访问审核中心
6. 活动负责人提交信息不完整的活动
7. 管理员驳回活动
8. 学生取消已报名活动
9. 已发布活动修改关键字段
10. 活动被取消后学生查看详情
