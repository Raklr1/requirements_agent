# 校园活动策划与报名管理平台 Product Brief

- 产品名称：校园活动策划与报名管理平台
- 产品形态：面向高校场景的 Web 平台
- 文档用途：为需求分析、后续设计、开发、测试 Agent 提供稳定的产品背景、目标、角色、范围和业务约束输入

## 2. 产品背景

高校中的讲座、比赛、志愿服务、社团活动、宣讲会等活动，通常存在以下问题：

1. 活动信息分散在微信群、QQ群、公众号、班群或线下海报中，学生难以统一获取。
2. 报名方式不统一，常见方式包括问卷、接龙、私聊登记，容易产生重复报名、漏登记和统计错误。
3. 活动发起、审核、发布之间缺少统一流程，活动负责人与审核老师之间沟通成本高。
4. 报名名单、人数上限、签到前名单确认等工作依赖手工处理，效率低且不透明。
5. 不同角色的信息视图不一致，学生、活动负责人、管理员都缺少清晰的操作入口。

因此，需要建设一个统一的平台，把“活动策划 -> 提交审核 -> 发布展示 -> 学生报名 -> 名单与统计管理”整合到一个可追踪、可管理、可扩展的系统中。

## 3. 产品愿景

建设一个适合高校日常使用的校园活动管理平台，使其能够：

1. 为学生提供统一、清晰、可搜索的活动入口。
2. 为活动负责人提供标准化的活动创建、编辑、提交和报名管理能力。
3. 为管理员提供审核、发布、驳回、统计和秩序管理能力。
4. 为后续系统扩展预留统一的数据结构和流程基础。

## 4. 产品目标

### 4.1 总体目标

实现一个支持多角色协作的校园活动管理系统，覆盖活动全生命周期管理，并能在课程项目范围内落地为可演示、可扩展的完整 Web 产品。

### 4.2 业务目标

1. 降低学生查找活动和完成报名的成本。
2. 降低活动负责人发布活动和维护报名名单的成本。
3. 让管理员能够用标准流程完成活动审核和发布管理。
4. 减少重复报名、超额报名、信息缺失、未审核直接发布等管理问题。

### 4.3 系统目标

1. 提供统一的账户与角色入口。
2. 提供清晰的活动状态流转机制。
3. 提供结构化的活动、报名、审核数据。
4. 提供满足后续测试和统计分析的可追踪记录。

## 5. 目标用户与角色

### 5.1 游客

- 可浏览已发布活动的公开信息。
- 不可创建活动、不可审核活动、不可提交正式报名。

### 5.2 学生用户

- 主要目标：浏览活动、筛选活动、查看详情、报名、取消报名、查看自己的报名记录。
- 典型活动：讲座、比赛、志愿服务、社团活动、文体活动。

### 5.3 活动负责人

- 身份包括：社团负责人、学生会成员、学院活动组织者、项目组织人。
- 主要目标：创建活动草稿、编辑活动、提交审核、查看审核结果、管理自己活动的报名情况。

### 5.4 管理员

- 身份包括：学院老师、平台管理员、学生工作管理人员。
- 主要目标：审核活动、发布活动、驳回活动、查看统计、维护平台秩序。

## 6. 核心业务流程

平台围绕以下主流程展开：

1. 用户注册并登录系统。
2. 学生或游客浏览已发布活动。
3. 活动负责人创建活动草稿并完善信息。
4. 活动负责人提交审核。
5. 管理员审核活动。
6. 审核通过后活动进入已发布状态并开放报名。
7. 学生报名活动，系统校验名额、重复报名和报名资格。
8. 活动负责人查看报名名单和人数。
9. 管理员查看平台统计和审核记录。

## 7. 功能范围

### 7.1 本期必须覆盖的功能范围

#### A. 账户与身份

1. 用户注册

1. 创建活动草稿

3. 提交活动审核

1. 查看待审核活动列表
2. 查看活动详情和关键字段
3. 审核通过

5. 查看已发布、已驳回、已下线活动

#### F. 统计与记录

1. 平台活动数量统计
2. 报名人数统计
3. 最近审核记录

## 8. 关键业务规则

### 8.1 角色与权限

1. 游客只能浏览公开活动信息。
2. 学生用户可以报名已发布活动，但不能审核活动。
3. 活动负责人可以管理自己创建的活动，但不能审核自己的活动。
4. 管理员可以审核和发布活动，但不能代替其他用户报名活动。

### 8.2 活动状态

活动至少包含以下状态：

1. `draft`：草稿，活动负责人可继续编辑。
2. `pending_review`：已提交审核，等待管理员处理。
3. `rejected`：审核未通过，活动负责人可查看驳回原因并修改后重新提交。
4. `published`：审核通过并对学生开放报名。
5. `registration_closed`：活动仍保留展示，但不再允许报名。
6. `completed`：活动已结束。
7. `cancelled`：活动被取消。

### 8.3 报名规则

1. 只有 `published` 状态的活动允许报名。
2. 当活动报名人数达到上限时，系统不得再接受新的报名请求。
3. 同一学生不能重复报名同一活动。
4. 学生在活动报名截止前可以取消报名，取消后名额应释放。
5. 已结束、已取消、未发布的活动不能报名。

### 8.4 活动信息完整性

活动在提交审核前，至少应具备以下字段：

1. 活动名称
2. 活动类型
3. 活动简介
4. 主办方
5. 活动地点
6. 活动开始时间
7. 活动结束时间
8. 报名截止时间
9. 人数上限
10. 面向对象/参与范围

### 8.5 审核规则

1. 信息不完整的活动不能审核通过。
2. 管理员驳回活动时必须填写驳回原因。
3. 已审核通过的活动如需修改关键字段，应重新进入审核流程。
4. 审核记录应保留审核人、审核时间、审核结论和说明。

## 9. 关键用户场景

### 9.1 学生场景

1. 学生注册并登录后，进入活动广场浏览近期活动。
2. 学生根据“讲座”“志愿服务”“比赛”等类型筛选活动。
3. 学生进入活动详情页，确认时间、地点、剩余名额和报名截止时间。
4. 学生填写必要信息后完成报名。
5. 学生在个人中心查看自己已报名的活动，并可在允许条件下取消报名。

### 9.2 活动负责人场景

1. 活动负责人登录后进入自己的工作台。
2. 创建一条新活动，填写完整信息后保存为草稿。
3. 核对信息后提交审核。
4. 若管理员驳回，则查看驳回原因并修改后重新提交。
5. 审核通过后查看活动报名人数和名单。

### 9.3 管理员场景

1. 管理员登录后台后查看待审核列表。
2. 打开活动详情核验信息是否完整和合理。
3. 对符合要求的活动执行通过操作。
4. 对不符合要求的活动执行驳回并说明原因。
5. 在后台查看活动总量、待审核数量和报名统计。

## 10. 关键数据对象

平台至少围绕以下核心对象建模：

1. 用户 `User`
2. 活动 `Event`
3. 报名记录 `Registration`
4. 审核记录 `ReviewRecord`
5. 通知/站内消息 `Notification`

## 11. 非功能目标

### 11.1 易用性

1. 普通学生应能在较少步骤内完成“浏览 -> 查看详情 -> 报名”流程。
2. 活动负责人应能明确看到活动当前状态和下一步可执行操作。
3. 管理员应能快速识别待处理活动并完成审核。

### 11.2 性能

1. 常用页面应在正常校园网络环境下较快加载。
2. 搜索、筛选、打开详情、提交表单等核心操作应具备可接受的响应速度。

### 11.3 可靠性

1. 对重复提交、超额报名、无权限操作、缺失字段等情况应给出明确反馈。
2. 关键操作应避免因重复点击造成重复数据。

### 11.4 可维护性

1. 系统应具备清晰的模块边界。
2. 后续可扩展签到、消息通知、活动推荐、数据分析等能力。

## 12. 本期成功标准

若系统达到以下结果，可认为本期产品目标基本达成：

1. 多角色主流程能够完整跑通。
2. 活动状态流转明确且可验证。
3. 学生能够完成报名和取消报名。
4. 活动负责人能够完成创建、提交审核和查看名单。
5. 管理员能够完成审核通过与驳回。
6. 系统能够展示基础统计与记录。
